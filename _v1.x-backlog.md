# v1.x Backlog — ai-orchestra

> Findings captured from the post-v1 pilot. Each item is a friendliness gap surfaced by running the universal orchestra against a real, high-existing-infra host project. None are blockers for v1; all are improvements that make the orchestra friendlier to projects that already have substantial agentic setup.

This file is project-agnostic — it captures **what the orchestra should improve**, not which project surfaced the issue. The pilot report that produced these findings lives outside the orchestra (on the host project that hosted the pilot run).

---

## How to read this file

Each finding has:

- **Priority** — relative ranking among v1.x items (high / medium / low).
- **Observation** — what the orchestra does today, neutrally described.
- **Proposal** — concrete fix to ship in v1.x.
- **Why** — what improves when this lands.
- **Touched files** — orchestra paths that would change.

Findings are listed in the recommended ship order.

When a finding ships, move its entry under the `## Shipped` section at the bottom with a short note linking to the changelog entry that closed it.

## Shipped

### F4 — Stop-hook conceptual overlap — shipped in 1.2.0

Closed by the v1.2.0 entry in [`CHANGELOG.md`](CHANGELOG.md). The orchestra now detects stop-hook overlap during Phase 3 inventory ([`core/discovery/existing-infra.md`](core/discovery/existing-infra.md) §3.11), surfaces the conflict in Phase 6 with a three-choice question form, records the chosen resolution in the install marker, and verifies it via §8.5 of every adapter's `post-install-checks.md`.

### F2 — Director rule double-load on suffix-rename — shipped in 1.3.0

Closed by the v1.3.0 entry in [`CHANGELOG.md`](CHANGELOG.md). Renamed copies of always-on rules are now downgraded to `alwaysApply: false` before writing, with a `[Orchestra — manual trigger]` description prefix and an explanatory body comment. Four new post-install checks verify the downgrade was applied across all four adapters.

### F7 — Mobile stack pack — shipped in 1.3.0

Closed by the v1.3.0 entry in [`CHANGELOG.md`](CHANGELOG.md). New `core/stack-packs/mobile/` pack with 4 rule files (touch-and-viewport, native-plugin-lifecycle, offline-and-sync, app-store-readiness), skills addenda, and roles addenda. Cross-link added to `core/discovery/signals/mobile.md`.

### F5 — Skill name overlap surfacing — shipped in 1.3.0

Closed by the v1.3.0 entry in [`CHANGELOG.md`](CHANGELOG.md). Suffix-renamed orchestra skills now carry `[Orchestra]` prefix in description and a disambiguation note. Post-install report includes `## Overlapping skills` section when any overlaps are detected.

### F1 — Multi-package / polyglot detection — shipped in 1.3.0

Closed by the v1.3.0 entry in [`CHANGELOG.md`](CHANGELOG.md). Secondary scan added to DETECTION.md §3.4 (depth-1 manifest detection, 20-entry cap, 1-second budget). `subProjects[]` field added to install schema. Post-install check §8.6 added to all four adapters.

### F3 — Always-on rule ceiling check — shipped in 1.3.0

Closed by the v1.3.0 entry in [`CHANGELOG.md`](CHANGELOG.md). Post-install check §8.7 warns when always-on context sources exceed 4. Adapted to each IDE's concept of "always-on". Non-blocking informational warning; agent lists each source.

### F6 — Pack rule glob filter — shipped in 1.3.0

Closed by the v1.3.0 entry in [`CHANGELOG.md`](CHANGELOG.md). Pack rules whose `## When this applies` globs match zero tracked files are skipped at install time and recorded in `stacks[].skippedPackRules[]`. All four adapters updated. Audit re-evaluates skipped rules on every run.
