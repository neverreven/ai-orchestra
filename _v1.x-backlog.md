# v1.x Backlog — ai-orchestra

> Findings captured from the post-v1 pilot. Each item is a friendliness gap surfaced by running the universal orchestra against a real, high-existing-infra host project. None are blockers for v1; all are improvements that make the orchestra friendlier to projects that already have substantial agentic setup.

This file is project-agnostic — it captures **what the orchestra should improve**, not which project surfaced the issue. The pilot report that produced these findings lives outside the orchestra (on the host project that hosted the pilot run).

---

## How to read this file

Each finding has:

- **Priority** — relative ranking among v1.x items (high / medium / low).
- **Observation** — what the orchestra does today, neutrally described.
- **Proposal** — concrete fix to ship in v1.x.
- **Why** — what improves when this lands.
- **Touched files** — orchestra paths that would change.

Findings are listed in the recommended ship order.

When a finding ships, move its entry under the `## Shipped` section at the bottom with a short note linking to the changelog entry that closed it.

## F4 — Stop-hook conceptual overlap (priority: high)

**Observation.** When a host project already has a stop hook in `.cursor/hooks.json` (or the IDE-equivalent location) that updates a learnings file, the orchestra's stop hook ALSO targets a learnings file. The merge logic in `adapters/<ide>/mappings.md` correctly preserves the existing hook and appends the orchestra's, so nothing is overwritten — but both hooks now fire on every session end and both append to the same learnings document. Two passes over the same conversation, possibly producing contradictory edits.

**Proposal.** Detect existing stop hooks whose `prompt` body references the same learnings path the orchestra would target. When detected, the adapter should NOT silently `merge-json`. Instead:

1. Surface the overlap in the dry-run plan as a `propose` action.
2. Offer three explicit choices to the user:
   - **(a)** Skip the orchestra hook (the existing project hook keeps updating the file).
   - **(b)** Replace the project hook with the orchestra hook.
   - **(c)** Tag the existing project hook with `metadata.orchestra: true` and adopt it as the orchestra's entry going forward.

Record the user's choice in the install marker so the audit can detect future drift.

**Why.** This is the highest-impact UX issue surfaced by the pilot. The current behaviour is correct (no overwrite) but produces silent duplicate work — exactly the kind of friction that erodes user trust on the second invocation.

**Touched files.** `core/discovery/existing-infra.md` (detect overlap), each `adapters/<ide>/mappings.md` (overlap branch in the merge rules), `core/registry/install.schema.md` (record user's choice), `adapters/<ide>/post-install-checks.md` (verify the choice was honoured).

## F2 — Director rule double-load on suffix-rename (priority: medium)

**Observation.** When a host project already owns a `.cursor/rules/ai-director.mdc` (or IDE-equivalent always-on rule with the same target path), the orchestra applies its `suffix-rename` conflict rule and writes its version under `<basename>.orchestra.<ext>`. The original is preserved — but the orchestra's renamed copy keeps `alwaysApply: true` in its frontmatter, so the IDE loads BOTH always-on Director rules into every session. The result is two competing always-on session protocols.

**Proposal.** When `suffix-rename` is the conflict action for any `alwaysApply: true` rule, downgrade the renamed copy's frontmatter to `alwaysApply: false` and add a manual-trigger description (e.g., "Run when you want to follow the orchestra's session protocol instead of the project's own."). Document the change clearly in the post-install report so the user understands why the orchestra version isn't always-on.

**Why.** Avoids silently overlaying two always-on rules. The user gets an explicit choice between the two protocols rather than both running by default.

**Touched files.** `adapters/<ide>/mappings.md` §6 (conflict-handling actions), `adapters/<ide>/render-rules.md` (frontmatter rules for renamed copies), `adapters/<ide>/post-install-checks.md` (verify renamed always-on rules are downgraded).

## F5 — Skill name overlap surfacing (priority: medium)

**Observation.** When the orchestra's universal skills collide by name with skills already present in the host project's skill folder (e.g., `cleanup`, `pre-release`), the adapter suffix-renames the orchestra version — both end up installed at slightly different paths. Trigger phrases overlap (e.g., "clean up the mess" matches both). The user is left to figure out from filenames which one fires.

**Proposal.** When `suffix-rename` triggers for a skill, the adapter should:

1. Read the existing project skill's `description` (the IDE matches on description text).
2. Adjust the orchestra skill's rendered `description` to add a disambiguation prefix or suffix (e.g., "Use this orchestra-managed skill when you want the universal version. The project also defines a same-named skill at `<path>` — read both and choose.").
3. Surface the overlap as a dedicated `## Overlapping skills` row in the post-install report, listing both descriptions side-by-side.

**Why.** The current behaviour is correct (no overwrite) but opaque. The user should not need to inspect filenames to understand which trigger fires.

**Touched files.** `adapters/<ide>/render-rules.md` (description-rendering rules), `adapters/<ide>/post-install-checks.md` (overlap reporting).

## F1 — Multi-package / polyglot detection (priority: medium)

**Observation.** The current discovery probe in `core/discovery/DETECTION.md` scans root-level signals only. Polyglot repositories with their own sub-package manifests (e.g., a separate backend under `server/`, a Rust desktop shell under `src-tauri/`, a mobile project under `android/`, a Python service under `backend/`) get correct but understated stack scores at the root level, and the sub-projects are entirely invisible to the install plan.

**Proposal.** Extend `core/discovery/DETECTION.md` with a "secondary scan" pass: after the root scan completes, look at top-level sub-directories that contain their own manifest file (`package.json`, `Cargo.toml`, `pyproject.toml`, `go.mod`, `pom.xml`, `build.gradle`, `*.csproj`, `sfdx-project.json`). Surface each as an entry in `profile.subProjects[]`. Stack packs remain root-level for v1.x; sub-project tagging is a profile field, not yet a pack-application axis.

**Why.** Without this, polyglot repos always get a slightly-wrong stack inventory, which propagates into the install marker and the audit's drift report — and into any future per-stack guidance the audit would offer.

**Touched files.** `core/discovery/DETECTION.md` (add secondary-scan procedure), `core/registry/install.schema.md` (add `subProjects[]` field), each `adapters/<ide>/post-install-checks.md` (verify sub-project enumeration when present).

## F7 — Mobile stack pack (priority: medium)

**Observation.** The discovery probe detects mobile (Capacitor, React Native, Flutter, MAUI) via `core/discovery/signals/mobile.md`, but v1 ships no mobile stack pack. Detection records the score in the install marker as info-severity drift; no rules, role addenda, or skill addenda are produced. Mobile patterns (touch targets, safe areas, offline-first auth, native plugin etiquette, app-store readiness) are genuinely useful and orthogonal to other packs.

**Proposal.** Ship a `core/stack-packs/mobile/` pack covering Capacitor + React Native + Flutter + MAUI patterns. Pack rules: touch targets and viewport, native plugin lifecycle, offline / pull-to-refresh patterns, push notifications, deep links, app-store submission checklists, store-readiness audit. Roles addenda extend `mobile-engineer.md` with stack-specific non-negotiables.

**Why.** Mobile + JS/TS is one of the most common polyglot configurations in 2026. Not having a pack means the orchestra can detect mobile but offers nothing actionable for it.

**Touched files.** New: `core/stack-packs/mobile/_overview.md` + `rules/*.md` + `skills.md` + `roles.md`. Update: `core/discovery/signals/mobile.md` (link to the pack).

## F3 — Always-on rule ceiling check (priority: low)

**Observation.** Each adapter writes 2 always-on rules (Director + orchestra-context). Stack packs are glob-scoped so they don't add to the count. Project-owned always-on rules (often 1–4 in mature codebases) plus orchestra rules can push the always-on count past 4 quickly. The IDE loads every always-on rule into every session — context-window pressure is real but invisible at install time.

**Proposal.** Add a post-install check that:

1. Counts the total number of always-on rules in the IDE's rule folder after install.
2. Emits a `warning`-severity entry in the post-install report when the count exceeds a soft threshold (suggested: 4).
3. Lists each always-on rule with its `description` so the user can decide which to demote to manual-trigger.

The check is informational, not blocking. The audit re-runs it on every audit invocation.

**Why.** Always-on rules are the cheapest context to break a session's signal-to-noise ratio. A check makes the cost visible to the user before it bites them.

**Touched files.** Each `adapters/<ide>/post-install-checks.md` (add the count check).

## F6 — Pack rule globs match nothing (priority: low)

**Observation.** Stack-pack rules are installed per-pack, not per-rule-applicability. A rule whose `## When this applies` globs match zero files in the project (for example, the JS/TS pack's TypeScript rule on a JS-only project, or the Python web pack's Django rule on a Flask-only project) is still written into the IDE's rule folder. It sits there matching nothing, cluttering the rule listing.

**Proposal.** Filter pack rules at install time by glob match against the project's tracked file set. If a rule's globs match zero files, the rule is **not installed**. The audit re-evaluates this on every run, so adding a matching file later re-installs the rule. The install marker records both installed and skipped pack rules so the audit can tell the difference between "filtered out" and "drift".

**Why.** Cleaner IDE rule listings; no dead rules cluttering the rule UI; a more honest install marker.

**Touched files.** Each `adapters/<ide>/mappings.md` §7 (pack rule application), `core/registry/install.schema.md` (add `stacks[].installedPackRules[]` and `stacks[].skippedPackRules[]`), each `adapters/<ide>/post-install-checks.md` (verify filter behaviour).

## F8 — Rust / Tauri stack pack (deferred to v2)

**Observation.** `core/discovery/signals/rust.md` detects Rust projects, with Tauri called out as a desktop-relevance signal. v1 ships no Rust pack. Tauri-rich projects (Rust backend + JS/TS frontend) are common in 2026 and would benefit from stack-specific patterns: cargo workspaces, error types, async runtimes, FFI safety, Tauri command/event boundaries.

**Decision.** Deferred to v2. Rust is a legitimate first-class candidate but the v1.x release should focus on UX friction (F1–F7) before broadening stack coverage. v2 would ship `core/stack-packs/rust/` with optional Tauri rules.

This finding is included in the v1.x backlog for visibility but is **not** scoped to v1.x.

## v2 candidates (out of v1.x scope, listed for boundary)

These items are already in the v1 close-out backlog (`CHANGELOG.md`, v1 close-out section). They're listed here so the v1.x scope has a clear boundary:

- **Multi-project orchestration runtime.** Read `~/.ai-orchestra/projects.json` and audit / upgrade every registered project from one place.
- **Distribution mechanism.** `npx ai-orchestra init` or equivalent — drop the orchestra into a new project without copying the folder by hand.
- **Audit auto-trigger via stop-hook.** A scheduled audit run instead of audit-on-demand. Cadence TBD (suggested: first session of a new day, or weekly).
- **Stack-pack depth.** Deepen the existing v1 packs (`js-ts`, `python-web`, `salesforce`) with framework-specific patterns — e.g., RSC patterns in Next, channels in Django, OmniStudio in Salesforce.
- **Non-Cursor adapter depth.** Bring `claude-code`, `codex`, `vscode` adapters to full parity with `cursor`.
- **Adversarial fixtures.** v1 fixtures cover the happy path and existing-infra preservation; v2 should add fixtures with deliberately broken metadata.
- **Self-extraction CLI.** Tooling to extract `ai-orchestra/` from a host repository into its own standalone repo while preserving the changelog and version history.
