# Cursor adapter — INSTALL.md

> **You are an AI agent.** You arrived here because [`RUN.md`](../../RUN.md) (Phase 4) routed you to the Cursor adapter after detecting Cursor as the IDE. This file is the Cursor-specific procedure. Read it top-to-bottom and follow it exactly.
>
> The Cursor adapter is a **full** v1 adapter. It satisfies every clause of [`../_contract.md`](../_contract.md) without gaps. If you find a clause you cannot satisfy in the actual project, stop and report — do not improvise.

---

## 1. What this adapter does

Translates the orchestra's project-agnostic content (rules, skills, Director, learnings, hooks, MCP slots) into Cursor's native locations:

- `.cursor/rules/*.mdc` — frontmatter-bearing rule files (always-on, glob-scoped, or manual).
- `.cursor/skills/<skill>/SKILL.md` — skill specs in their own folders.
- `.cursor/hooks.json` — start / stop / other lifecycle hooks.
- `.cursor/mcp.json` — Model Context Protocol server registry.
- `AGENTS.md` (project root) — consolidated project context with an orchestra-managed section.
- `_documentation/AI_LEARNINGS.md` (or detected equivalent) — living learnings document.
- `.ai-orchestra/install.json` — install marker per [`../../core/registry/install.schema.md`](../../core/registry/install.schema.md).
- `~/.ai-orchestra/projects.json` — optional global registry append.

The detailed mapping is in [`mappings.md`](mappings.md). The shape of each rendered file is in [`target-schema.md`](target-schema.md).

## 4. Walkthrough — what to do per RUN.md phase

### Phase 1 — IDE detection (already done)

By the time you read this, RUN.md detected Cursor and selected this adapter. You do not need to re-detect. Confirm via these signals (any one is sufficient):

- Environment variable `CURSOR_TRACE_ID` is set.
- Process tree includes `Cursor.exe` / `cursor` binary.
- `.cursor/` already exists in the project (strong but not required).

If none of these are observed in the runtime, ask the user before proceeding.

### Phase 2 — Discovery (already done)

Phase 2 of RUN.md produced the project profile. You do not re-run discovery here. Just keep the profile available — `mappings.md` references it (e.g., for stack-pack inclusion).

### Phase 3 — Existing-infra inventory (Cursor-specific check)

Beyond the generic existing-infra inventory in [`../../core/discovery/existing-infra.md`](../../core/discovery/existing-infra.md), perform these Cursor-specific checks:

- List every `.mdc` file in `.cursor/rules/`. Capture each file's frontmatter (`description`, `alwaysApply`, `globs`). Treat anything not generated by a previous orchestra run as **user-owned** and never overwrite it.
- List every folder in `.cursor/skills/`. For each, read its `SKILL.md`'s frontmatter `name` and treat the skill as user-owned unless the install marker says otherwise.
- Read `.cursor/hooks.json` if present. Note the existing hook entries; the adapter must preserve all of them and **add** the orchestra's stop-hook (or merge with an existing stop-hook entry — see [`mappings.md`](mappings.md) §5).
- Read `.cursor/mcp.json` if present. Note all `mcpServers[*]` entries.
- Read `AGENTS.md` if present. Search for the canonical marker pair (see [`mappings.md`](mappings.md) §3). If present, the adapter extends the section; if absent, it appends a new managed section at end-of-file.

Record everything found in the install plan as `existing` so the user sees what is being preserved.

### Phase 4 — Adapter loaded (this file)

You are here. Move on once you have read this entire file once.

### Phase 5 — Build the install plan (dry-run)

Produce the plan strictly per [`mappings.md`](mappings.md). The plan must:

1. Have a deterministic order — alphabetical by target path within each artifact category. Idempotent re-runs depend on this.
2. Include a `source` field on every entry (which orchestra core file produced it).
3. Mark `action` per the conflict-handling table in [`mappings.md`](mappings.md) §6.
4. Include a `gaps` array. For the **full** Cursor adapter in v1, this array MUST be empty. If you find a real gap during the run, that is a bug in this adapter — record it but do not silently proceed.
5. Include the MCP slot list per [`mcp.md`](mcp.md).
6. Include the registry write payload per [`target-schema.md`](target-schema.md) §7.

### Phase 6 — Confirm

Per RUN.md. Wait for `apply` / `apply but skip ...` / `revise` / `abort`. The Cursor adapter has no special override.

### Phase 7 — Apply

Apply in this **strict order** (idempotency depends on it):

1. Create `.cursor/` and subfolders (`rules/`, `skills/`) if absent.
2. Write rule files (`.mdc`). For each:
   - If `action: create` — write the rendered template per [`render-rules.md`](render-rules.md).
   - If `action: skip` — verify identical to template; do nothing.
   - If `action: suffix-rename` — write to `<basename>.orchestra.mdc`, leave the original alone, surface the rename in the post-install report.
3. Write skill folders + `SKILL.md` files. Same action set as rules.
4. Render and write `AGENTS.md` (or merge the managed section into an existing one). Use the marker convention in [`mappings.md`](mappings.md) §3.
5. Render and write the learnings document (default `_documentation/AI_LEARNINGS.md`). If a learnings doc already exists at the resolved path, merge missing seed sections rather than overwriting.
6. Update `.cursor/hooks.json` (merge the orchestra's stop-hook entry into the existing JSON, preserving all other entries). See [`mappings.md`](mappings.md) §5.
7. Update `.cursor/mcp.json` per [`mcp.md`](mcp.md). Always write minimum-permission, slot-prefixed entries; never expand permissions on existing entries.
8. Write `.ai-orchestra/install.json` per [`target-schema.md`](target-schema.md) §7.
9. Append/update entry in `~/.ai-orchestra/projects.json`. If write is blocked, skip silently and record the skip.

If any step fails, **stop**. Report the partial state. Do not roll back automatically — the user decides next steps. Cursor's own version control (and the project's git status) makes recovery straightforward.

### Phase 8 — Post-install verification

Read [`post-install-checks.md`](post-install-checks.md) and run every check. Report results in the format described there. Any check failure is a critical problem to surface to the user.

### Phase 9 — Activation message

Activation in Cursor is **automatic**:

- New rules in `.cursor/rules/` are picked up by Cursor's rule engine on the next session.
- The stop-hook in `.cursor/hooks.json` fires the next time a session ends.
- The MCP entries in `.cursor/mcp.json` register with Cursor's MCP runtime on next start.

Tell the user, in the closing message:

- The exact list of `.cursor/` files written or modified.
- That the orchestra's stop-hook will fire at the end of the next conversation in this project.
- How to invoke the audit skill: "Ask the agent to `audit AI infra` (or any of the trigger phrases in [`../../core/skills/audit/ai-infra-audit/SKILL.md`](../../core/skills/audit/ai-infra-audit/SKILL.md))."
- Any MCP slots that were registered but require the user to attach a real server (per the slot list in [`mcp.md`](mcp.md)).
- Whether a global registry entry was created or updated at `~/.ai-orchestra/projects.json`. If this is the first install on the machine, mention that multi-project skills (`multi-project-audit`, `upgrade-all`) are now available.

## 5. Idempotency in Cursor specifically

A second invocation on an already-installed project must produce **only `skip` actions** when nothing has changed. The Cursor adapter achieves this via:

- Hash-comparing each rule / skill file against the template-rendered content.
- Stable JSON keys in `.cursor/hooks.json` and `.cursor/mcp.json` (sorted alphabetically when the adapter writes).
- The marker's `last_install` timestamp does NOT change on idempotent re-runs; the audit timestamp is the one that updates.

If a re-run would produce non-`skip` actions, surface them as a normal install plan. The user may have edited a rendered file by hand — the adapter respects that per the conflict-handling rules.

The Cursor adapter is the orchestra's reference implementation. v1 satisfies every clause of [`../_contract.md`](../_contract.md):

- ✅ Global registry — `~/.ai-orchestra/projects.json` append.

If during a real run a gap is discovered, that is a bug in this adapter to be filed against the next orchestra patch version. The adapter MUST report it; it MUST NOT silently downgrade.
